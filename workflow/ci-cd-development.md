# CI/CD 与线上开发环境

本文说明：**PR 合并到 `main` 后自动跑 CI，并在通过后更新线上开发环境**的流程意图与 GitHub 侧配置要点。工作流文件路径、job 名称等**事实源**以本仓库 `docs/`、CI 配置与实际实现为准（避免双处重复细节）。

## 1. 期望行为

1. 贡献者按 [git-and-review.md](./git-and-review.md) 从功能分支提 PR。
2. **PR 合并到 `main`** 在 GitHub 上表现为对 `main` 的一次 **push**。
3. 该 **push** 触发本仓库配置的 **CI 工作流**：运行团队约定的门禁（如静态检查、测试、契约/schema 漂移校验等）。
4. 门禁全部通过后，若工作流中包含**部署到开发环境**的步骤（常见为仅 `push` 到 `main` 时触发），则将构建结果发布到团队约定的环境。

**说明**：部署步骤中的具体命令由团队按基础设施填写；需与运维约定一致。

## 2. GitHub 仓库配置（建议）


| 配置项                           | 说明                                                                                                                                 |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **Environment（如 `development`）** | 在 **Settings → Environments** 创建环境；将部署密钥放在该环境的 **Secrets** 中。Workflow 中通过 `environment` 引用。 |
| **分支保护**                      | 对 `main` 开启 **Required status checks**，要求 CI 通过后再允许合并。                                                 |
| **Secrets 管理**                | 使用 Environment 级 secrets；若需审批再部署，可在该环境开启 **Required reviewers**。                                                    |


## 3. 与本地文档的关系

- 本地联调、`.env`、各服务端口以本仓库 `docs/` 与各服务文档为准。
- **线上环境的 URL、主机名、证书**由基础设施决定；宜记在运维文档或受控配置中，避免与代码推导文档混写导致漂移。

## 4. 后续扩展（可选）

- 可在同一或独立 workflow 中增加前端构建、E2E 等 job，再让部署 job 依赖它们。
- 详细命令行与 workflow 片段应维护在与对应模块同层的事实源文档中，避免治理文档重复实现细节。
