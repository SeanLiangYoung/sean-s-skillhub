# 工程契约与 AI 辅助开发原则（通用）

本文提炼**可复用**的工程化模式与用 AI 写代码时的纪律，**不**绑定特定产品、目录或脚本名。字段级定义以各仓库代码与 `docs/` 为准。

## 1. 可复用模式

### 单一事实源与分层文档

- **代码**：实现中的严格模型（如 Pydantic、OpenAPI 生成物等）宜作为对外/对内行为的权威定义（含未知字段策略等）。
- **契约说明**：专题文档集中在 `docs/`（或仓库约定目录），由**入口索引**串联，避免在编辑器规则里复制大段正文，减少双源漂移。
- **流程与范围**（非传输层契约）：人在回路、环境假设、验收口径等，可与端点契约分文档维护，降低误读。

### 自动化漂移门禁

- **模式**：若仓库提供「schema 导出 / 示例 JSON / 索引链接」等一致性校验脚本，契约变更后应跑通；CI 与本地合并前自检应对齐。

### 可观测与排错

- **模式**：跨服务联调约定关联标识；JSON 错误体宜含可机读错误码与同一关联标识；管线结构化日志的阶段名与文档表一致。

## 2. AI 辅助开发的纪律

- **文档优先**：动手前从仓库入口文档进入专题说明并对照实现中的 schema，**禁止**在未核对的情况下假设字段名、HTTP 形状或默认行为。
- **变更闭环**：若改动对外契约、默认值或示例，须同步**实现 + 文档 + 机器可读产物（若有）**，并跑通仓库规定的漂移或校验步骤。
- **完成标准**：以脚本与 CI 通过为准，不以模型自述为准；大段契约放在 `docs/` 并通过链接引用。

## 3. 分阶段落地（建议）


| 阶段    | 目标           | 产出物                        | 完成定义（示例）     |
| ----- | ------------ | -------------------------- | ------------ |
| P0 立项 | 划定契约层与流程层    | 索引 + schema 位置 + 服务说明入口   | 新成员能指到事实源   |
| P1 基线 | 对外请求体可校验     | 严格模型 + 示例 + 导出 schema（若有） | 示例通过严格校验    |
| P2 文档 | 人与自动化同读一套话   | 专题文档 + 先读后写、禁止臆造字段      | 契约类 PR 带文档更新 |
| P3 CI | 风格、行为、契约一起守 | Lint + 测试 + 漂移类检查           | 与主分支 CI 一致   |
| P4 可观测 | 联调可复盘        | 关联 ID、稳定 code、阶段名表       | 故障能凭 ID 串日志  |
| P5 迭代 | 需求 → 实现 → 不漂移 | 变更含 schema / 文档 / 日志节点     | 本地与 CI 等价自检  |


### 多人场景下的补充标准

- **Onboarding**：新成员能在约定时间内从索引找到事实源并跑通本地检查命令。
- **无单点口头约定**：关键集成点可查文档 + 代码 + 关联标识。
- **责任清晰**：明确契约 Owner（或轮值），对「是否算契约变更」能裁决。

## 4. 日常小循环（个人）

1. 打开索引 → 读相关专题文档与 schema 源码。
2. 改代码；需要时用 AI 辅助，但约束在已存在契约内。
3. 若动契约：同步文档与示例，运行仓库规定的校验脚本。
4. 提交前：按仓库文档与 [git-and-review.md](./git-and-review.md) 与 CI 对齐自检。
