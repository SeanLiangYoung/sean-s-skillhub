# 以项目为维度的多人协同

本文描述**多人、多角色**在同一仓库上协作时的边界与习惯，可与 [engineering-principles.md](./engineering-principles.md) 中的通用原则配合使用。

## 1. 项目边界


| 层级    | 建议维护方式                 | 多人如何对齐                                                        |
| ----- | ---------------------- | --------------------------------------------------------------- |
| 项目契约  | 指定 **契约 Owner**（可轮换）   | 对外 JSON/API、配置 schema 等以 **代码中的严格模型 + 专题文档** 为准；不在私聊里约定新字段。 |
| 项目索引  | Tech Lead 与契约 Owner 共识 | 单一入口链到各专题（常见名：`AGENTS.md`、`README`）；新成员只跟索引走。                 |
| 运行与流程 | 产品/交付 Owner（可与技术分立）    | 人在回路、环境、验收标准写在流程类文档，避免与传输层契约混在同一篇里误读。                             |


**原则**：一个业务方向 = 一套可追溯契约 + 一个索引 + 明确 Owner；契约是否变更由 Owner 或团队约定快速裁决。

## 2. 角色（可兼任，但职责不悬空）

- **契约 Owner**：schema、示例、对外文档、漂移检查与 CI 门禁含义；契约类 PR 须满足其关注点（或书面代理）。
- **管线 / 可观测 Owner**（可合并）：关联标识、管线阶段名表、日志字段；保证跨服务、跨人排障能串联。
- **项目协调**：里程碑与范围；避免「对外已承诺而仓库未更新」。
- **贡献者**：在索引与 schema 约束内开发；大改动先做小范围设计或契约评审再扩写。

## 3. 异步协作习惯

- **排障**：请求与响应统一带团队约定的**关联标识**（及 HTTP 头等）；工单或聊天中粘贴该标识。
- **决策留痕**：契约或默认行为变更落在 **PR + 对应文档段落**，讨论结论不长期只留在即时消息里。

## 4. 冲突预防

- **先读后写**：认领任务时先看索引与相关专题，再看实现中的模型；新人 onboarding 以索引路径为第一课。
- **禁止口头 API**：各端之间的新字段须先出现在严格模型与文档中，再写联调代码。

## 5. 与迭代节奏的结合（可选）


| 节奏   | 内容                                        |
| ---- | ----------------------------------------- |
| 迭代开始 | 对齐本迭代是否触及契约；若触及，为契约 Owner 留出评审时间。         |
| 迭代中  | 短同步：谁在动契约、谁阻塞在联调；阻塞优先补关联标识与文档链接。            |
| 迭代结束 | 验收以「可预览 + 契约与 CI 仍绿」为准；可选质量项按团队单独约定。       |
