# Git 分支、本地开发与 Pull Request

本文约定：**每人使用独立功能分支**，在**本地**（可用 AI 辅助）完成一个功能后 **提交 PR**，合并前通过检查与评审。

## 1. 分支策略

- **主分支**：`main`（或团队统一保护分支），**只通过 PR 合并**，不在其上长期直接开发。
- **个人功能分支**：从最新 `main` 拉出，命名建议能辨认责任人或意图（示例：`feature/<简述>`、`fix/<简述>`）。
- **粒度**：一条分支尽量对应**一个功能或一类修复**，避免单个 PR 混杂多件不相干事。

## 2. 本地开发（含 AI 辅助）

1. **开工前**：`git fetch`，检出并更新 `main`，再 `git checkout -b feature/<你的分支名>`。
2. **先读事实源**：与契约、API、配置模型相关的改动，先读**本仓库入口文档**与 `docs/` 中专题说明，并对照实现中的 schema，再让 AI 生成或修改代码，降低「幻觉字段」。
3. **提交**：开发过程中小步 `commit`，便于 diff 与回溯。
4. **推 PR 前**：在仓库根执行与 **CI 等价**的自检（具体命令以本仓库 CI 工作流或 `README` / `docs/` 为准）。

若改动涉及契约或对外示例，还须同步文档与示例，并确认仓库规定的漂移或 schema 校验通过。

## 3. 提交 Pull Request

1. 推送分支：`git push -u origin feature/<你的分支名>`。
2. **PR 描述**建议包含：做什么、为何做（可链到 issue）；是否 **契约/API/文档变更**；需要 reviewer 重点看的点。
3. **合并条件**：CI 通过 + 团队规定的评审；**不在 CI 失败时合并**。

## 4. 多人并行

- 定期将 `main` 合并进自己的分支（或按团队约定 rebase），减少巨大冲突。
- **大改动先沟通**：动公共契约、管线阶段名、HTTP 形状时，在 PR 中 `@` 契约 Owner 或先短会。

## 5. AI 与「完成」的边界

每人自己的分支是个人工作区；**合并进 `main` 即团队共同事实**。本地可用 AI 加速实现，但 **合并门槛仍以检查通过 + 评审 + 契约/文档一致** 为准。若使用 AI 生成大量代码，可在 PR 中简要注明依据的文档或 commit。
